# AI Council Decision — Post-L4.63 Synthesis

**Operator:** Bartek · **Date:** 2026-06-08 · **Lead architect:** Claude Opus (primary builder) · **Mode:** planning/research only (human implements)

## 1. Current state assessment

You have a genuinely capable single-persona Telegram operator with real scaffolding — 312 passing tests, a live Windows listener, delivery cards, a 5-level risk model, budget guard + kill switch, cost ledger with correct reserved→final collapse, read-only connectors for Gmail/Calendar/Drive/GitHub, gated provider executors, and an FTS5 memory store. That is real and ahead of a toy. But measured against the goal ("Poke + Hermes memory + OpenClaw hands"), **parity is not done, and three load-bearing pillars are partially hollow.**

Verified facts from the code (not the brief):

- **Conversation memory is NOT durable in production.** `state/conversations.jsonl` is physically **absent** from the live state dir, while `costs.jsonl` (406KB), `actions.jsonl` (68KB) and `errors.jsonl` (25KB) all exist from the same running listener. The cause is in code: `append_conversation_turn(user)` and `append_conversation_turn(assistant)` both run at the **success-path tail** of `listen_once` (ai_council.py ~14867-14868), *after* send, *after* routing — so the router's `recent_conversation()` lookup for the next turn has nothing to read. The Poke headline feel ("text it, it remembers, follow up naturally") **does not work** right now. This is the #1 reality gap.
- **The LLM router (L4.63) is off by default and unsafe to turn on as-is.** With the router on, far more free-form text reaches Grok, whose free-text `prompt` is re-routed; the *only* fence against an R3/R4 side-effect riding an allowlisted command into an auto-started R0 background task is `risk_level_for_text()` (line 7693) — a substring matcher with **confirmed English gaps**: "remove", "wipe", "transfer money", "pay", "invoice", "deploy", "prod", "send email to <x>" are NOT in its deny sets. There is **zero regression test** at the `route_message` boundary asserting the router can't leak a destructive intent.
- **No durable user-fact memory exists.** All 14 `memory_save` call sites write machine work-products (`kind` ∈ artifact/recipe/action/project_memory/council_report/task). **Nothing** captures a fact the user *states* in chat. So `memory_context_for_prompt` never surfaces "I prefer short answers" / "my flight is Tuesday." This is the Hermes/mem0 core that's missing.
- **OpenClaw "hands" are still a promise.** No true local filesystem executor; workspace writes are bounded to 20KB prompt artifacts; the Codex worker is double-gated and approval-only. Genuine local action does not exist yet.
- **Integrations are read-cached + gated-write only.** No autonomous recipe-driven workflow; every write needs manual approval behind `AI_COUNCIL_PROVIDER_WRITE_ENABLED=false`.
- **Durability/concurrency is not 24/7-grade.** Batch-level offset commit + non-atomic `write_offset` + unlocked `append_jsonl` shared across listener and detached workers = replay storms and interleaved-write corruption on the Windows box where it actually runs.

**Verdict:** strong substrate, but the "feels like one assistant with memory" property is currently an *illusion held up by short transcript windows that aren't even persisting*. The next sprint must make memory real AND make the about-to-be-enabled router safe — these are the same code paths.

## 2. Top 3 gaps by impact

**Gap 1 — Conversation memory is not durably persisting; follow-ups can't use context. (Impact: HIGH)**
`conversations.jsonl` is absent on the live host; turns are written post-send at the loop tail, so routing turn N+1 sees nothing, and a send failure or crash loses the user turn entirely. *Why it matters:* this is the single property that separates "one continuous operator" from "stateless bot." It is also the prerequisite for the router (L4.63) to deliver natural follow-ups. Three of four council members independently flagged this as their #1 or #2.

**Gap 2 — The L4.63 router has no risk fence; enabling it can leak destructive intents. (Impact: HIGH)**
`risk_level_for_text()` is an English-gappy keyword denylist with no boundary regression test. Turning on the router (the host's intended next action) without this fence risks an R3/R4 side-effect auto-executing via an allowlisted command. *Why it matters:* an agent with gated hands is only as safe as the gate that decides what's gated; a silent leak here is the highest-severity failure mode in the whole system.

**Gap 3 — No durable user-fact recall ("text it a fact, it remembers"). (Impact: HIGH)**
The memory store never captures user-stated facts/preferences/commitments, so cross-session recall — Poke's and Hermes's headline differentiator — does not exist. *Why it matters:* without it, the product is a good chat router, not a personal assistant with a relationship. This is the defining feature of the goal.

> **Decisive call:** Gaps 1 and 2 are the *same files and the same blast radius*, and both are hard blockers on safely turning the router on. Gap 3 is the marquee feature but is **higher-risk to ship first** (memory poisoning via unverified LLM extraction) and structurally *depends on* durable conversation persistence existing first. So the correct order is: **fix the substrate (1+2) now, then build the feature (3) on top.**

## 3. Chosen next sprint after L4.63

### L4.64 — Durable Conversation Memory + Router-to-Risk Leak Fence

**(adopts the Safety Officer's sprint as the spine, folds in the Infra Engineer's exactly-once/atomic-write hardening as required sub-tasks)**

**Why this and why now:** It makes the headline Poke follow-up feature *actually function* (turns persist before routing, so "a teraz zrób z tego plan" can see prior context) AND installs the gate that lets the host safely flip `AI_COUNCIL_LLM_ROUTER=true`. Both council scores here were the joint-highest (88 safety, 86 infra). It is read-only/local-only, no new external surface, and it unblocks the entire roadmap. Shipping Gap-3 user-fact memory *before* this would build durable recall on top of a store that isn't even persisting and a router that can leak — strictly worse sequencing.

**Acceptance criteria:**

1. **Persist-before-route.** Move the inbound USER-turn `append_conversation_turn` to run **before** `route_message()` in `listen_once`, and wrap user+assistant persistence so an exception in `build_response`/send can never silently skip the user-turn write (try/finally or early persist). Routing turn N+1 can read turn N.
2. **E2E persistence test.** A test driving the real listener handler for two sequential messages on one `chat_id` asserts `conversations.jsonl` contains turn 1 *before* turn 2 is routed; with the router mocked, turn 2's routing prompt provably includes turn 1's text.
3. **Router-to-risk fence test.** For a representative destructive/external set (PL + EN: "delete all files", "wyślij maila do klienta", "pay the invoice", "publish to prod", "transfer money", plus one obfuscated variant), assert `route_message()` with the router enabled+mocked **never** returns a command that auto-starts a background R0 task carrying an R3/R4 prompt — it must fall back to `/chat` or a draft/approval path. Test must fail on a deliberately weakened matcher (proves it bites).
4. **Close the keyword gaps.** Extend `risk_level_for_text()` to cover the confirmed English holes ("remove", "wipe", "send email to", "transfer money", "pay", "invoice", "deploy", "prod") with an assertion per addition. No change loosens any existing classification.
5. **Atomic offset write.** `write_offset` writes to a temp file in `STATE_DIR` then `os.replace` onto `OFFSET_FILE` (mirror the existing `CONTROL_FILE` pattern at ~4710-4712). `read_offset` on a missing/empty file falls back to the last known processed update_id, **not** None (no full-backlog replay).
6. **Per-update offset commit + replay dedup.** Commit the offset per fully-handled update (not once per batch), and consult a durable processed-update set at the top of the per-update loop so a replay skips routing/persistence/nudges for already-handled updates. `append_conversation_turn` is idempotent per update (no duplicate turns on replay); a replayed task-creating update resolves to the existing `task_id` (extend idempotency beyond the 90s window).
7. **Locked `append_jsonl`.** Serialize cross-process appends with a **per-file** `BlockingFileLock` (reuse existing primitive) so listener + detached workers can't interleave bytes in `conversations.jsonl`/`tasks.jsonl`/`progress_events.jsonl`. **Fail-safe:** on lock timeout, append anyway with a recorded warning — never drop a row. Stress test: two processes append 500 rows each → 1000 valid JSON lines, zero `read_jsonl` skips.
8. **State-dir discipline + ledger isolation.** `CONVERSATIONS_FILE` honors `AI_COUNCIL_STATE_DIR` so tests never touch live state; extend the L4.63 conftest md5-isolation check to `conversations.jsonl`.
9. **Health visibility.** `/health` surfaces conversation-memory liveness (`last_turn_at` / `conversation_turns_today`), offset value, processed-set size, and whether the last offset committed atomically.
10. **Docs + green suite.** `docs/implementation/L4_64_DURABLE_MEMORY_AND_ROUTER_RISK_FENCE.md` written, stating the router MUST stay off until this lands and listing the exact host smoke checks. Full Mac suite green (≥312 + new tests); `py_compile` clean.

## Underweighted risks

- **Stale-evidence risk is the meta-risk and is unweighted.** The synthesis was built on a "verified" premise that isn't reproducible, and the operator's own bootstrap doc says "data >48h old: re-check." If the host code drifted from what the council reviewed, L4.64's surgical edits (reorder persistence @14867, atomic write_offset @12208) could target moved/renamed code. Add a "re-pin line numbers on host before edits" gate.
- **Memory-poisoning risk in L4.65 is named but under-mitigated.** Gated LLM extraction of user facts is the single highest-blast-radius feature (a poisoned "fact" silently steers every future turn). The roadmap gives it provenance + supersede, but no human-confirm-before-trust step, no quarantine for low-confidence extractions, and no eval that adversarial input can't write a durable false fact. For an ADHD operator who won't audit memory, a wrong durable fact is worse than no memory.
- **Builder>seller drift is an underweighted strategic risk.** The operator's memory flags this explicitly and repeatedly; seven more infra layers materially increase it. The risk isn't a bug — it's shipping a beautiful substrate that still earns 0 PLN. This belongs in the risk register, not just the gaps doc.
- **Windows/NTFS locking is correctly flagged as load-bearing but the fail-safe is self-contradictory.** "On lock timeout, append anyway with a warning — never drop a row" reintroduces exactly the interleaved-write corruption the lock exists to prevent. Under contention you get both: a warning AND a corrupted line. The risk of the mitigation itself is unweighted; needs a bounded retry + per-line atomic write (write to temp, fsync, rename or O_APPEND single-write ≤PIPE_BUF) rather than "append anyway."
- **Single-host SPOF / no backup-restore drill.** One Windows box runs everything; the deploy checklist says "snapshot state\ before deploy" but there's no tested restore path. A corrupted memory.sqlite or offset with no rehearsed recovery = silent data loss for the one feature (memory) the whole roadmap is about.
- **DND / proactivity-safety is unaddressed for L4.71.** The bootstrap doc mandates "DND 23:00–07:00 CET — no proactive pings unless CRITICAL." Proactive wake (L4.71) with no quiet-hours/criticality gate risks 3am pings — a trust-killer. Not in any acceptance criterion.
